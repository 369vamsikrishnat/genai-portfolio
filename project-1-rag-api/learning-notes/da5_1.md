# 1. Introduction to LangChain

## What is LangChain?

LangChain is an open-source framework that simplifies building applications powered by Large Language Models (LLMs).

Instead of writing everything from scratch, LangChain provides reusable components that help developers build AI applications such as:

- Chatbots
- Document Question Answering Systems
- RAG Applications
- AI Agents
- Multi-Agent Systems
- Workflow Automation

Think of LangChain as the **backend framework for LLM applications**, similar to how FastAPI simplifies backend API development.

---

## Why was LangChain Created?

LLMs alone have several limitations.

They:

- Cannot access your private documents.
- Cannot search databases by themselves.
- Cannot remember long conversations reliably.
- Cannot call external tools without additional logic.

LangChain provides reusable building blocks that solve these problems.

It helps developers connect LLMs with:

- PDFs
- Word Documents
- CSV Files
- SQL Databases
- APIs
- Vector Databases
- External Tools

---

## Why Use LangChain?

Without LangChain, developers must manually:

- Read files
- Split documents
- Generate embeddings
- Store vectors
- Perform similarity search
- Build prompts
- Call the LLM
- Parse responses

LangChain standardizes these operations into reusable components.

---

Without LangChain:


With LangChain:

```
↓

↓

↓

↓

↓

```

Each component has a dedicated responsibility.

---

# Core Components

LangChain consists of multiple independent components.

| Component | Purpose |
|-----------|---------|
| Document Loader | Reads data from external sources |
| Text Splitter | Splits large documents into chunks |
| Embedding Model | Converts text into vectors |
| Vector Store | Stores embeddings |
| Retriever | Finds relevant chunks |
| Prompt Template | Creates prompts dynamically |
| Chat Model | Generates the final answer |

Each component can be replaced independently.

For example:

- Replace OpenAI with Anthropic.
- Replace Chroma with Pinecone.
- Replace PDF Loader with Web Loader.

The rest of the pipeline remains unchanged.

# 2. Documents

## What is a Document?

In LangChain, a **Document** is the standard data structure used to represent information.

Instead of passing raw text throughout the pipeline, LangChain wraps the content inside a `Document` object.

A document contains two parts:

1. **page_content** – The actual text.
2. **metadata** – Additional information about the text.

Example:

```python
Document(
    page_content="LangChain is an open-source framework for LLM applications.",
    metadata={
        "source": "langchain.pdf",
        "page": 5
    }
)
```

---

## Why Not Use Plain Strings?

Instead of:

```python
text = "LangChain is an open-source framework..."
```

LangChain uses:

```python
Document(
    page_content=text,
    metadata={...}
)
```

This allows the framework to keep track of:

- Source file
- Page number
- Author
- URL
- Creation date
- Chunk ID
- Any custom metadata

---

## Document Structure

```
Document
│
├── page_content
│      │
│      └── Actual text
│
└── metadata
       │
       ├── source
       ├── page
       ├── author
       └── custom fields
```

---

## Why Metadata Matters

Metadata is **not sent to the embedding model as the document text**, but it is extremely useful after retrieval.

Example:

A user asks:

> "What does Page 12 say about insurance claims?"

Retriever returns:

```
page_content:
Insurance claims...

metadata:
page = 12
source = Insurance.pdf
```

The application can now display:

> Source: Insurance.pdf (Page 12)

Without metadata, you would know the answer but not where it came from.

---

## Common Metadata Fields

| Field | Purpose |
|--------|----------|
| source | File name or URL |
| page | PDF page number |
| author | Document author |
| title | Document title |
| chunk_id | Unique chunk identifier |
| created_at | Creation date |

You can also add custom metadata depending on your application.

---

## Interview Questions

### Q1. What is a Document in LangChain?

**Answer:**

A Document is LangChain's standard object for storing text along with its metadata. It contains `page_content` and a `metadata` dictionary.

---

### Q2. Why is metadata important?

**Answer:**

Metadata helps identify where the retrieved information came from, making it useful for citations, filtering, debugging, and displaying document sources.

---

# 30-Second Revision

A Document is the basic unit of information in LangChain. It stores both the actual text (`page_content`) and descriptive information (`metadata`). Using Document objects instead of plain strings allows LangChain to preserve important context such as file names, page numbers, authors, and other useful attributes throughout the RAG pipeline.

# 3. Document Loaders

## What is a Document Loader?

A Document Loader is responsible for reading data from external sources and converting it into LangChain `Document` objects.

Think of it as the **entry point** of the RAG pipeline.

Without a loader, LangChain has nothing to process.

---

## Why Do We Need Document Loaders?

Different data sources have different formats.

For example:

- PDF
- Word (.docx)
- CSV
- TXT
- HTML
- Websites
- SQL Databases

Each requires different logic to extract text.

LangChain provides loaders so developers don't have to write parsing logic themselves.

---

## Loading Pipeline

```
PDF
 │
 ▼
PyPDFLoader
 │
 ▼
Document Objects
 │
 ▼
Text Splitter
```

---

## Common Document Loaders

### PyPDFLoader

Reads PDF files page by page.

```python
from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader("sample.pdf")
documents = loader.load()
```

Each page becomes a separate `Document`.

---

### TextLoader

Reads plain text files.

```python
from langchain_community.document_loaders import TextLoader

loader = TextLoader("notes.txt")
documents = loader.load()
```

---

### CSVLoader

Loads CSV files into Documents.

Useful for:

- Customer data
- Sales reports
- Inventory

---

### WebBaseLoader

Loads content directly from websites.

```python
loader = WebBaseLoader("https://example.com")
documents = loader.load()
```

Useful for:

- Documentation
- Blogs
- Knowledge bases

---

## Output of a Loader

Regardless of the source, loaders return a list of `Document` objects.

```
PDF

↓

Loader

↓

[
 Document(...),
 Document(...),
 Document(...)
]
```

This standardization allows the rest of the pipeline to remain the same.

---

## Interview Questions

### Q1. What is the purpose of a Document Loader?

**Answer:**

A Document Loader reads data from external sources and converts it into LangChain `Document` objects for further processing.

---

### Q2. Does every loader return the same type?

**Answer:**

Yes. Regardless of the data source, loaders return a list of `Document` objects.

---

# Common Mistakes

❌ Assuming loaders generate embeddings.

❌ Assuming loaders split text automatically.

❌ Thinking loaders store documents in a vector database.

A loader only **reads and converts data into Document objects**.

---

# 30-Second Revision

Document Loaders are the starting point of the LangChain pipeline. They read data from various sources such as PDFs, text files, CSVs, or websites and convert the content into standardized `Document` objects. This allows the rest of the pipeline to process information consistently, regardless of where it came from.

# 4. Chat Models

## What is a Chat Model?

A Chat Model is the component responsible for generating responses from a Large Language Model (LLM).

LangChain provides wrappers around different LLM providers, allowing you to use a common interface regardless of the underlying model.

Examples:

- OpenAI
- Anthropic Claude
- Google Gemini
- Mistral
- Llama

---

## OpenAI vs ChatOpenAI

One of the most common beginner questions is:

> What is the difference between OpenAI and ChatOpenAI?

### OpenAI

OpenAI is the company that provides AI models such as:

- GPT-4.1
- GPT-4o
- GPT-5 (when available)
- Embedding models

Think of **OpenAI** as the AI service provider.

---

### ChatOpenAI

`ChatOpenAI` is a **LangChain class**.

Its job is to communicate with OpenAI's chat models while following LangChain's standard interface.

Example:

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    model="gpt-4o",
    temperature=0
)
```

Internally,

```
Your Code

↓

ChatOpenAI

↓

OpenAI API

↓

GPT Model

↓

Response
```

ChatOpenAI is **not** the model.

It is simply a wrapper around the OpenAI API.

---

## Why Use ChatOpenAI?

Imagine changing from GPT to Claude.

Without LangChain:

You rewrite large parts of your application.

With LangChain:

Only one line changes.

```python
ChatOpenAI(...)
```

↓

```python
ChatAnthropic(...)
```

Everything else stays the same.

---

Most LangChain components use:

```python
invoke()
```

Example

```python
response = llm.invoke("What is LangChain?")
```

Output

```python
AIMessage(

```

```python
response = llm.invoke(question)
```

```
AIMessage

├── content
├── metadata

```

Usually we use

```python
response.content
```

to get the final text.

### HumanMessage

### AIMessage

# 5. Prompt Templates

## What is a Prompt Template?

A Prompt Template allows you to create prompts dynamically by inserting variables into predefined text.

Instead of writing prompts manually every time, you define a template once and reuse it.

---

## Why Use Prompt Templates?

Without Prompt Templates:

```python
question = "What is RAG?"

prompt = f"""
Answer the following question:

{question}
"""
```

This quickly becomes difficult to maintain.

With Prompt Templates:

```python
prompt = PromptTemplate.from_template(
    "Answer the following question:\n\n{question}"
)

prompt.invoke({
    "question": "What is RAG?"
})
```

The template stays the same.

Only the variables change.

---

## PromptTemplate

Used for text-based prompts.

Example:

```python
from langchain_core.prompts import PromptTemplate

prompt = PromptTemplate.from_template(
    "Explain {topic} in simple terms."
)

prompt.invoke({
    "topic": "Vector Databases"
})
```

Output:

```
Explain Vector Databases in simple terms.
```

---

## ChatPromptTemplate

Used when working with chat models.

Instead of a single string, it creates structured messages.

Example:

```python
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an AI tutor."),
    ("human", "{question}")
])
```

After rendering:

```
System:
You are an AI tutor.

Human:
Explain RAG.
```

This is the preferred approach when using chat-based models.

---

## Prompt Rendering

```
Template

↓

Variables

↓

Rendered Prompt

↓

Chat Model
```

---

## PromptTemplate vs ChatPromptTemplate

| PromptTemplate | ChatPromptTemplate |
|----------------|--------------------|
| Plain text | Chat messages |
| Simple prompts | Multi-message conversations |
| Usually for text completion | Used with chat models like GPT and Claude |

---

## Why Templates Matter

Benefits:

- Reusable
- Cleaner code
- Easier maintenance
- Dynamic prompts
- Less duplication

Instead of changing prompt text everywhere, update the template once.

---

## Interview Questions

### Q1. What is a Prompt Template?

**Answer:**

A Prompt Template is a reusable prompt containing variables that are filled dynamically at runtime.

---

### Q2. Why use ChatPromptTemplate?

**Answer:**

Because chat models expect structured messages such as system, human, and AI messages rather than a single block of text.

---

### Q3. What is the difference between PromptTemplate and ChatPromptTemplate?

**Answer:**

PromptTemplate generates plain text prompts, while ChatPromptTemplate generates structured chat messages suitable for chat-based LLMs.

---

# Common Mistakes

# 30-Second Revision

Prompt Templates allow developers to create reusable, dynamic prompts by replacing variables at runtime. `PromptTemplate` is used for plain text prompts, while `ChatPromptTemplate` creates structured chat messages for chat models. Using templates improves code readability, maintainability, and consistency across AI applications.

# 6. Embedding Models

## What are Embeddings?

An embedding is a numerical representation (vector) of text.

Instead of understanding words directly, machine learning models work with numbers. Embeddings convert text into vectors while preserving **semantic meaning**.

For example:

```
"I love dogs."

↓

[0.23, -0.18, 0.91, ...]
```

The numbers themselves are not meaningful to humans, but the relationship between vectors is meaningful to the model.

---

## Why Do We Need Embeddings?

Large Language Models can read text, but they cannot efficiently search millions of documents.

Embeddings solve this problem by representing text mathematically.

Instead of comparing text word by word, we compare vectors.

Example:

```
Text

↓

Embedding Model

↓

Vector

↓

Vector Database

↓

Similarity Search
```

---

## How Embeddings Work

Suppose we have two sentences.

```
The cat is sleeping.

The kitten is resting.
```

Although the wording is different, both sentences have a similar meaning.

After embedding:

```
Sentence A

↓

[0.12, 0.44, ...]

Sentence B

↓

[0.10, 0.46, ...]
```

The vectors are very close together because the meanings are similar.

Now compare:

```
The cat is sleeping.

↓

[0.12, 0.44, ...]

I want to buy a car.

↓

[-0.81, 0.03, ...]
```

These vectors are far apart because the meanings are different.

This is called **semantic similarity**.

---

## Semantic Similarity

Traditional keyword search looks for matching words.

Example:

Search:

```
Vehicle
```

Document:

```
Car
```

Keyword search may fail because the exact word "vehicle" is missing.

Embedding search succeeds because it understands that **car** and **vehicle** are semantically related.

---

## OpenAIEmbeddings

LangChain provides wrappers for embedding models.

Example:

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small"
)
```

The embedding model converts text into vectors.

---

## Embedding Pipeline

```
Raw Text

↓

Embedding Model

↓

Vector

↓

Vector Store
```

---

## Query Embeddings

Not only documents are embedded.

The user's question is also converted into a vector.

Example:

```
Documents

↓

Embeddings

↓

Stored

---------------------

User Question

↓

Embedding Model

↓

Query Vector

↓

Similarity Search
```

Both documents and queries exist in the same vector space.

This allows similarity comparisons.

---

## Important Point

Embeddings are usually generated **once** during indexing.

User queries are embedded **every time a search is performed**.

Example:

```
Offline

PDF

↓

Embedding

↓

Store

---------------------

Online

Question

↓

Embedding

↓

Search
```

---

## Interview Questions

### Q1. What is an embedding?

**Answer:**

An embedding is a numerical vector representation of text that captures semantic meaning, allowing similarity comparisons.

---

### Q2. Why do we need embeddings?

**Answer:**

Embeddings allow semantic search by converting text into vectors that can be compared mathematically.

---

### Q3. Are document embeddings generated every time?

**Answer:**

No.

Document embeddings are usually generated once during indexing and stored.

Only user queries are embedded at runtime.

---

❌ Thinking embeddings store knowledge.

# 30-Second Revision

Documents are embedded during indexing, while user queries are embedded at runtime so both can be compared in the same vector space.

# 7. Vector Stores

## What is a Vector Store?

A Vector Store is a specialized database that stores embeddings and performs similarity search.

Instead of storing plain text for searching, it stores vector representations of documents.

---

## Why Do We Need a Vector Store?

Imagine storing one million document embeddings.

Searching every vector manually would be extremely slow.

A Vector Store builds indexes that make similarity search efficient.

Without a Vector Store:

```
Query

↓

Compare with Doc 1

↓

Compare with Doc 2

↓

Compare with Doc 3

↓

...

↓

Compare with 1,000,000 Docs
```

Very slow.

With a Vector Store:

```
Query

↓

Vector Index

↓

Nearest Neighbours

↓

Top Results
```

Much faster.

---

## Popular Vector Stores

### Chroma

- Open source
- Easy to use
- Excellent for local development
- Common in tutorials

---

### FAISS

Developed by Meta.

Advantages:

- Extremely fast
- Runs locally
- Ideal for similarity search

---

### Pinecone

Cloud-hosted vector database.

Advantages:

- Managed service
- Highly scalable
- Production-ready

---

## Vector Store vs Vector Database

People often use these terms interchangeably.

Generally:

Vector Store:
A component or library used to store and retrieve vectors.

Vector Database:
A full database system built specifically for vector search.

Examples:

- Chroma
- Pinecone
- Weaviate
- Milvus

---

## Similarity Search

Suppose a user asks:

```
How does RAG work?
```

Steps:

```
Question

↓

Embedding

↓

Vector

↓

Similarity Search

↓

Top 4 Documents

↓

Returned
```

The Vector Store returns the documents whose vectors are closest to the query vector.

---

## Architecture

```
Chunks

↓

Embeddings

↓

Vector Store

↓

Similarity Search

↓

Relevant Documents
```

---

## Interview Questions

### Q1. What is a Vector Store?

**Answer:**

A Vector Store stores embeddings and performs efficient similarity searches to retrieve relevant documents.

---

### Q2. Why not store plain text?

**Answer:**

Similarity search works on vectors, not raw text. Vectors capture semantic meaning, enabling more accurate retrieval.

---

### Q3. Name some popular Vector Stores.

**Answer:**

- Chroma
- FAISS
- Pinecone
- Weaviate
- Milvus

---

# Common Mistakes

❌ Thinking the Vector Store generates embeddings.

❌ Thinking it generates answers.

❌ Confusing a Vector Store with an LLM.

---

# 30-Second Revision

A Vector Store stores document embeddings and performs efficient similarity searches. When a user asks a question, the query is converted into an embedding, compared against stored vectors, and the most semantically similar documents are returned. Popular Vector Stores include Chroma, FAISS, and Pinecone.

# 8. Retrievers

## What is a Retriever?

A Retriever is responsible for finding the **most relevant documents** for a user's query.

It sits between the Vector Store and the LLM.

The LLM **does not search** the Vector Store directly.

Instead, the Retriever searches on behalf of the LLM.

---

## Why Do We Need a Retriever?

A Vector Store knows how to store vectors.

A Retriever knows **how to retrieve useful information**.

Think of it this way:

```
Vector Store
    │
    ▼
Stores Embeddings
```

```
Retriever
    │
    ▼
Searches & Returns
Relevant Documents
```

A Retriever is an abstraction built on top of the Vector Store.

---

## Retrieval Flow

```
User Question
      │
      ▼
Embedding Model
      │
      ▼
Query Vector
      │
      ▼
Retriever
      │
      ▼
Vector Store
      │
      ▼
Top Matching Documents
```

Notice that the LLM is **not involved** during retrieval.

The Retriever finishes its work **before** the LLM receives any context.

---

## Creating a Retriever

Most Vector Stores provide:

```python
retriever = vector_store.as_retriever()
```

Now instead of calling the Vector Store directly:

```python
docs = retriever.invoke(question)
```

The Retriever handles the search.

---

## Search Types

### Similarity Search

Returns the documents with the highest similarity score.

Example:

```
Top 4 Most Similar Chunks
```

---

### Similarity Search with Score

Returns both:

- Documents
- Similarity Score

Example:

```
Document A
Score: 0.92
```

Useful for filtering weak matches.

---

### Maximum Marginal Relevance (MMR)

Sometimes the top results are almost identical.

Example:

```
Chunk 1

Chunk 2

Chunk 3

Chunk 4
```

All may come from the same paragraph.

MMR tries to increase diversity.

Instead of:

```
Chapter 5

Chapter 5

Chapter 5

Chapter 5
```

It returns

```
Chapter 5

Chapter 2

Chapter 8

Chapter 11
```

This gives the LLM broader context.

---

## Retriever vs Vector Store

| Retriever | Vector Store |
|-----------|--------------|
| Searches documents | Stores vectors |
| Returns Documents | Stores embeddings |
| Built on top of Vector Store | Database component |
| Used during querying | Used during indexing & querying |

---

## Interview Questions

### Q1. What is a Retriever?

**Answer:**

A Retriever is a component that searches a Vector Store and returns the most relevant documents for a user's query.

---

### Q2. Does the Retriever generate answers?

**Answer:**

No.

It only retrieves relevant documents.

The LLM generates the final answer.

---

### Q3. Why use a Retriever instead of querying the Vector Store directly?

**Answer:**

A Retriever provides a standardized interface for searching and supports configurable retrieval strategies such as similarity search and Maximum Marginal Relevance (MMR).

---

# Common Mistakes

❌ Thinking the Retriever generates embeddings.

❌ Thinking the Retriever calls the LLM.

❌ Thinking the Retriever stores vectors.

---

# 30-Second Revision

A Retriever is responsible for searching a Vector Store and returning the most relevant documents for a user query. It does not generate answers or embeddings—it simply finds useful context that will later be passed to the language model.

# 9. Indexing Pipeline

## What is Indexing?

Indexing is the **offline process** of preparing documents for retrieval.

It converts raw documents into searchable vector representations.

Users are **not involved** during indexing.

The indexing process usually happens once when documents are added or updated.

---

## Why Do We Need Indexing?

LLMs cannot search PDFs directly.

Documents must first be transformed into embeddings and stored.

Without indexing:

```
PDF

↓

LLM
```

Impossible.

With indexing:

```
PDF

↓

Loader

↓

Text Splitter

↓

Embeddings

↓

Vector Store
```

Now the documents are searchable.

---

## Complete Indexing Pipeline

```
Raw Documents
      │
      ▼
Document Loader
      │
      ▼
Document Objects
      │
      ▼
```

## Interview Questions

### Q1. What is indexing?

**Answer:**

Indexing is the offline process of converting documents into vector embeddings and storing them for efficient retrieval.

---

### Q2. Why is indexing performed offline?

**Answer:**

Generating embeddings for every document is computationally expensive. Doing it once during indexing makes user queries much faster.

---

### Q3. What is the difference between indexing and retrieval?

**Answer:**

Indexing prepares and stores searchable document embeddings offline. Retrieval searches those stored embeddings at runtime to find relevant context for a user's query.

# 30-Second Revision

The indexing pipeline prepares documents for semantic search by loading them, splitting them into chunks, generating embeddings, and storing those embeddings in a Vector Store. This happens offline and only when documents are added or updated. During runtime, the Retriever searches these stored vectors to find relevant context for the LLM.
