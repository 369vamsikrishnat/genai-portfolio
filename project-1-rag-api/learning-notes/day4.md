# Vector Search & Pinecone - Revision Notes

These are my personal revision notes for Vector Search, ANN, HNSW, SQL databases, and Pinecone indexing algorithms.

---

# 1. What is Nearest Neighbor Search?

The goal of nearest neighbor search is to find the vectors that are most similar to a given query vector.

In RAG systems, documents are converted into **embeddings**, which are high-dimensional numerical vectors generated by an embedding model.

Example:

```
Document
    ↓
Embedding Model
    ↓
[0.21, 0.45, ..., 0.83]
```

When a user asks a question:

```
Question
    ↓
Embedding Model
    ↓
Query Vector
```

The vector database searches for the stored vectors that are closest to the query vector.

Similarity is commonly measured using:

* Cosine Similarity
* Dot Product
* Euclidean Distance

---

# 2. Exact Nearest Neighbor Search

Exact search compares the query vector against **every vector** stored in the database.

```
Query Vector

↓

Compare with Vector 1
Compare with Vector 2
Compare with Vector 3
...
Compare with Vector N

↓

Return the closest vectors
```

### Advantages

* 100% accurate
* Always returns the true nearest neighbors

### Disadvantages

* Very slow for large datasets
* Time complexity is approximately **O(N)**

If a database contains 10 million vectors, all 10 million must be compared.

---

# 3. Approximate Nearest Neighbor (ANN)

Approximate Nearest Neighbor (ANN) speeds up similarity search by avoiding comparisons with every stored vector.

Instead, it intelligently searches only the most promising regions.

```
Query Vector

↓

Search promising regions

↓

Return vectors that are approximately nearest
```

### Advantages

* Extremely fast
* Scales to millions or billions of vectors
* Usually achieves 95–99% recall
* Used in almost every production RAG system

### Trade-off

Small loss in accuracy for a massive gain in speed.

---

# 4. What is HNSW?

**HNSW (Hierarchical Navigable Small World)** is one of the most popular ANN algorithms.

Instead of storing vectors in a simple list, HNSW organizes them into a **multi-layer graph**.

```
Top Layer
      ●
     / \
    ●   ●

Middle Layer
 ●──●──●──●

Bottom Layer
●─●─●─●─●─●
```

### Search Process

1. Start at the top layer.
2. Make large jumps toward the target.
3. Move down layer by layer.
4. Refine the search.
5. Return the nearest vectors.

Think of it like Google Maps:

* Highway → Main Road → Street → House

Instead of checking every house, you quickly navigate toward the destination.

### Advantages

* Very fast
* High recall
* Excellent for large vector datasets

Commonly used by:

* Qdrant
* Weaviate
* Milvus
* Faiss (HNSW index)

# 5. Why Can't a Normal SQL Database Do This Efficiently?

Traditional SQL databases are optimized for structured queries such as:

```sql
SELECT *
FROM users
WHERE age > 25;
```

They use indexes like:

* B-tree
* Hash
* Bitmap
* GiST (in some databases)

These indexes are designed for:

* Equality lookups
* Range queries
* Sorting
* Joins

They are **not designed for high-dimensional similarity search**.

An embedding may contain hundreds or thousands of dimensions:

```
[0.23,
0.91,
-0.44,
...
1536 values]
```

To find similar vectors, a traditional SQL database often has to compare the query against every stored vector.

This becomes extremely slow as the dataset grows.

---

# 6. How Vector Databases Solve This Problem

Vector databases use ANN indexing algorithms instead of traditional database indexes.

Workflow:

```
Documents

↓

Embedding Model

↓

Embeddings

↓

Vector Database

↓

ANN Index

↓

Top-K Similar Results
```

Instead of scanning every vector, the ANN index efficiently navigates toward the most similar vectors.

---

# 7. Does Pinecone Use HNSW?

**No.**

This is a common misconception.

Although Pinecone has educational articles explaining HNSW and other ANN algorithms, its production serverless architecture does **not** use HNSW internally.

Instead, Pinecone automatically selects different indexing algorithms depending on the size of each **slab** (storage unit).

| Slab Size                   | Index Used                                          |
| --------------------------- | --------------------------------------------------- |
| Small (≈ up to 10K vectors) | Ananas                                              |
| Medium (≈10K–100K vectors)  | PQFS (Product Quantization Fast Scan)               |
| Large (>100K vectors)       | IVF + PQFS                                          |
| Newly inserted vectors      | Exact Brute Force Search (until indexing completes) |

This adaptive indexing strategy is one of Pinecone's key architectural optimizations.

---

# 8. Why Does Pinecone Teach HNSW Then?

Pinecone's learning center explains many ANN algorithms used across the industry, including:

* HNSW
* IVF (Inverted File Index)
* Product Quantization (PQ)
* DiskANN
* Locality Sensitive Hashing (LSH)

These articles are intended to teach vector search concepts and are **not** descriptions of Pinecone's internal implementation.

---

# 9. Interview Takeaways

Remember the following:

* Embeddings represent documents as high-dimensional vectors.
* Similarity search retrieves the closest vectors.
* Exact search compares every vector and is slow.
* ANN speeds up search by exploring only promising regions.
* HNSW is a graph-based ANN algorithm widely used in vector databases.
* Traditional SQL databases are not optimized for high-dimensional similarity search.
* Pinecone uses proprietary adaptive indexing (Ananas, PQFS, IVF + PQFS) instead of HNSW.

# Key Terms

* **Embedding** – Numerical vector representation of data.
* **Similarity Search** – Finding vectors most similar to a query vector.
* **Exact Search** – Compares against every stored vector.
* **ANN** – Approximate Nearest Neighbor search for fast retrieval.
* **HNSW** – Graph-based ANN indexing algorithm.
* **IVF** – Inverted File Index for partitioning vectors.
* **PQ (Product Quantization)** – Compresses vectors to reduce memory and speed up search.
* **PQFS** – Product Quantization Fast Scan, used by Pinecone.
* **Ananas** – Pinecone's indexing algorithm for smaller datasets.
* **Vector Database** – A database optimized for storing and searching embeddings.
